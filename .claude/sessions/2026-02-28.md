## 주요 결정 및 근거

### `as any` 대신 module augmentation
`SharedConfig`의 `singleton` 속성이 플러그인 타입에서 주석 처리되어 있음.
`as any` / intermediate variable 모두 타입 안전성 없음이 빌드 테스트로 확인됨.
→ `declare module` 블록으로 인터페이스 병합. export되지 않은 interface도 모듈 네임스페이스 수준에서 병합 가능함을 실증.

```ts
declare module '@originjs/vite-plugin-federation' {
  interface SharedConfig { singleton?: boolean; eager?: boolean; strictVersion?: boolean }
}
```

## 학습 내용

- **bootstrap 패턴**: MF 공유 모듈 협상 완료 후 React 기동을 보장하기 위한 동적 import 패턴. "bootstrap"은 컴퓨팅의 bootstrapping 개념에서 유래 (Twitter Bootstrap UI 라이브러리도 동일 어원).
- **모듈 네임스페이스**: TypeScript가 모듈 단위로 타입을 격리하는 공간. `declare module 'pkg'`로 해당 네임스페이스를 열어 interface를 병합할 수 있음.
- **Phase 1 vs Phase 2 핵심 차이**:
  - Module Federation: 빌드 도구 의존, remoteEntry.js 런타임 fetch, shell은 dev 가능
  - Import Map: 브라우저 네이티브, 모든 앱 build+preview 필요, bootstrap 패턴 불필요
